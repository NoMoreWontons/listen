## Problem Statement
`listen` shipped homework upload, quiz/test generator, and YouTube intake (commit `4097010`). Scope is widening from "lecture recorder" toward a general study tool. Question: what do the popular dedicated study apps (Anki, Quizlet, RemNote) have that `listen` doesn't, and which of it is worth building.

## Landscape (Phase 2.75 — search, not competitive deep-dive)
- **Anki**: gold standard spaced-repetition scheduler (now FSRS), huge customization, manual card creation, dated UI.
- **RemNote**: notes ARE flashcards — highlight a line, it becomes a spaced-repetition card automatically. SM-2 default, FSRS optional.
- **Quizlet**: easiest onboarding, AI-generates flashcards/quizzes from pasted notes or docs, but no real SM-2 scheduling — "Learn mode" only approximates it.
- Common thread across all three 2026 write-ups: **spaced repetition scheduling is the one feature every popular tool centers on**, and it's the one thing `listen` has zero of. `listen`'s quiz generator is one-shot (take it once, get a score) — nothing comes back to you later at the right interval.

## What Makes This Cool
`listen` already has something none of Anki/Quizlet/RemNote start with: real lecture audio, auto-transcription, and an Obsidian graph already organized by semester → class → unit. Competitor tools make you manually write or import notes first. Here, the raw material to generate cards already exists the moment a lecture is filed.

## Premises
1. Spaced repetition is the highest-leverage gap — agree/disagree?
2. This is a single-user tool; collaboration/sharing/multi-device sync (Quizlet's social layer) is out of scope — agree/disagree?
3. Reuse existing patterns (Haiku call shape from `generate_quiz`, `quizzes` table shape, `_set`/raw-body upload conventions) rather than adding new frameworks — agree/disagree?

## Approaches Considered

### A. Flashcards + spaced repetition (SM-2, not FSRS)
- New `cards` table: `id, class, unit, front, back, due_at, interval_days, ease, created_at`.
- `POST /cards/generate` — same Haiku-call shape as `generate_quiz`, scoped by class/unit, reads filed summaries, emits `[{front, back}]`.
- `GET /cards/due` — cards where `due_at <= now()`.
- `POST /cards/{id}/review` body `{grade: again|hard|good|easy}` → SM-2 update of `interval_days`/`ease`/`due_at`.
- UI: a "Review" card in the side column showing due count, flip-card review loop (reuse quiz-taking UI's focus-guard pattern so it survives `refresh()`).
- ponytail: SM-2, not FSRS — FSRS needs a review-history dataset to tune; SM-2 is ~20 lines and good enough for one user. Upgrade path: swap the interval formula in `review()` if reviews start feeling mistimed.

### B. Anki `.apkg` export
- Skip building a review UI at all — export generated cards as a real `.apkg` (sqlite3 + media zip, no new dependency, stdlib `sqlite3` + `zipfile`) and let Anki's mature FSRS scheduler do the reviewing.
- Much less code than A, but review history lives outside `listen` — score/streak tracking (below) can't see it.

### C. Study streak / score history
- `quizzes.score` already exists — a `GET /stats` endpoint aggregating score-over-time and a review-streak counter needs no new table.
- Cheap add-on regardless of A vs B.

### D. Cheat-sheet export
- One-page markdown/PDF per unit built from summaries already filed to Obsidian. Pure formatting, no new data.

### E. Spaced-review due-date reminders
- `assignments_ics` already generates `.ics` for syllabus due dates — same code path could emit a calendar event for "N cards due today."

## Recommended Approach
**A (flashcards, SM-2)** first. It fills the one gap every popular competitor centers on, and it's a straight extension of code that already exists (`generate_quiz`'s Haiku-call shape → `cards`, `quizzes` table shape → `cards` table). B (Anki export) is worth keeping as a fallback if A's review UI turns out to be more effort than it's worth — could ship both (generate in `listen`, review in either place) without much extra cost since B is just a different serializer on top of the same `cards` table. C and D are cheap, do anytime. E is a two-line extension of existing `.ics` code, bolt on after A ships.

## Open Questions
- SM-2 vs FSRS: SM-2 chosen for simplicity; revisit only if review timing feels off in practice.
- Card generation granularity: per-unit only, or also per-recording? (per-unit matches existing quiz scoping, start there)

## Next Steps
1. `cards` table migration (same manual-apply pattern as `quizzes` — schema.sql + user runs it in Supabase SQL editor).
2. `POST /cards/generate`, `GET /cards/due`, `POST /cards/{id}/review` in `app.py`, SM-2 math as a pure function (testable like `grade_mcq`).
3. Review UI in `index.html`, outside `#list`/`#quizArea` refresh containers.
4. `test_cards.py` — pure unit test of the SM-2 interval function (no DB, no Claude), matching `test_quiz.py`'s style.
5. C/D/E as smaller follow-on tasks once A ships.
